# SPECIFICATION — Спецификация требований

> Документ описывает функциональные и нефункциональные требования, ограничения, критерии приёмки и use-case сценарии системы White-sheet-breaker.

---

## 1. Функциональные требования

 ID | Требование | Приоритет |
|---|---|---|
| 1.1 | Система должна позволять пользователю сформулировать правило в формате «ЕСЛИ [сигнал] → ТО [действие]» | Обязательный |
| 1.2 | Система должна запрашивать у пользователя цель (target) — файл, над которым выполняется действие | Обязательный |
| 1.3 | Система должна автоматически определять тип действия на основе цели | Обязательный |
| 1.4 | Система должна создавать новый файл с шаблоном или открывать существующий файл | Обязательный |
| 1.5 | Система должна поддерживать кроссплатформенное открытие файлов (Windows, macOS, Linux, WSL) | Обязательный |
| 1.6 | Система должна отслеживать активность пользователя по изменению размера файла | Обязательный |
| 1.7 | Система должна показывать меню помощи уровня 1 через 1 минуту бездействия | Обязательный |
| 1.8 | Система должна показывать меню помощи уровня 2 через 3 минуты бездействия | Обязательный |
| 1.9 | Система должна завершать сессию через 5 минут бездействия | Обязательный |
| 1.10 | Система должна адаптировать меню помощи под тип файла (.py, .md, .json, .yaml, .txt) | Обязательный |
| 1.11 | Система должна работать параллельно с Pomodoro-таймером | Обязательный |
| 1.12 | Система должна позволять пользователю выбрать длительность Pomodoro (1-50 минут) | Обязательный |
| 1.13 | Система должна отображать обратный отсчёт с прогресс-баром | Обязательный |
| 1.14 | Система должна логировать каждый выполненный ритуал в формате NDJSON | Обязательный |
| 1.15 | Система должна отправлять xAPI-стейтменты в LRS (mock или real режим) | Обязательный |

| ID | Ограничение | Обоснование |

## 4. Критерии приёмки (Acceptance Criteria)

### Полный цикл работы

```gherkin
GIVEN пользователь запустил модуль
WHEN он создал правило "ЕСЛИ файл пуст → ТО напишу класс → classes.py"
AND подтвердил правило
THEN должен создаться файл classes.py с python-шаблоном
AND должен запуститься Pomodoro-таймер (если выбран)
AND должно запуститься фоновое наблюдение
AND должен быть создан лог в NDJSON
AND должен быть отправлен xAPI-стейтмент
```

### Фоновое наблюдение с таймером

```gherkin
GIVEN пользователь выбрал Pomodoro-таймер на 25 минут
WHEN запущены таймер и наблюдение параллельно
THEN таймер должен работать независимо от наблюдения
AND при бездействии 1 минута должно появиться меню уровня 1
AND при выборе шаблона он должен вставиться в файл
AND таймер НЕ должен останавливаться
AND после завершения таймера наблюдение не должно остановиться
```

### Автозавершение при активности

```gherkin
GIVEN запущено фоновое наблюдение
WHEN пользователь добавил >100 символов в файл
THEN должно появиться сообщение "Работа началась"
AND наблюдение должно завершиться
```

## 5. Use-Case спецификация

### Создание правила с нуля

**Актор:** Пользователь

**Предусловие:** Модуль запущен

**Постусловие:** Правило сформулировано, файл создан/открыт

**Основной сценарий:**

1. Пользователь выбирает "Создать правило с нуля"
2. Система запрашивает сигнал (ситуацию)
3. Пользователь вводит сигнал
4. Система запрашивает действие
5. Пользователь вводит действие
6. Система запрашивает режим (создать/открыть)
7. Пользователь выбирает режим
8. Система запрашивает цель (имя файла)
9. Пользователь вводит имя файла
10. Система валидирует имя файла
11. Система показывает правило для подтверждения
12. Пользователь подтверждает
13. Система выполняет микро-шаг
14. Система запускает наблюдение (и опционально таймер)
15. Система логирует результат и отправляет xAPI-стейтмент

**Альтернативные сценарии:**

- 16. Файл уже существует (в режиме "create") → система запрашивает другое имя
- 17. Файл не найден (в режиме "open") → система запрашивает правильный путь
- 18. Пользователь отклоняет правило → возврат к шагу 2 (до 3 попыток)
